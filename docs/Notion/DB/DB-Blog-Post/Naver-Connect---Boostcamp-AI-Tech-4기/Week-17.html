Cloud

Cloud 서비스를 사용하는 이유

  • 웹, 앱 서비스를 만드는 경우
    • 자신의 PC로도 서비스를 만들 수 있음
    • 그러나 내 IP를 누구나 접근할 수 있게 하고 공유해야 함
    • 내 PC를 종료할 시, 웹과 앱 서비스로 종료됨
  • Local
    • Local에서 서비스 구축 시 전통적인 접근 방법
      • 물리적 공간, 확장성을 고려한 서비실을 만들고 운영
      • IDC(Internet Data Center) 공간 개설 및 운영
      • 서버를 넣을 공간 + 추후 서버를 추가할 떄 즉각적으로 확장할 수 있는지
      • 서버가 갑자기 종료되지 않도록 준비 필요
    • Local의 문제
      • 갑작스러운 트래픽 증가시 대응의 어려움
      • 너무 적은 트래픽일 때, 서버를 놀게 놔둬야 함
  • Cloud
    • 개방자가 직접 설정해야 했던 작업 등을 클라우드에서 쉽게 할 수 있는 방향으로 발전
    • Apache Spark를 쉽게 운영할 수 있도록 AWS EMR, GCP Dataproc 등을 활용
      • 직접 하둡을 설치할 필요 없이 이미 설치되어 있음
    • Jupyter Notebook 환경도 미리 설치해두고 사용할 수 있음

Cloud 서비스의 다양한 제품

  • 다양한 Service

    https://www.whatap.io/ko/blog/9/

    • IAAS(for IT administrators)
      • Amazon Web Service(AWS), Microsoft Azure, DigitalOcean, Google Compute Engine(GCE)
    • PAAS(for SW Developers)

    • Server Instance 생성 후 Instance에 들어가서 사용

  • Serverless Computing

    • Computing service와 유사, 그러나 서버 관리를 클라우드 쪽에서 진행
    • Code를 Cloud에 제출하면 그 코드를 가지고 서버를 실행해주는 형태
    • 요청 부하에 따라 자동으로 확정(Auto Scaling)
    • Micro Service로 많이 활용(ex: AWS Lambda)

    • Docker를 사용한 Container 기반으로 서버 실행
    • Docker Image를 업로드하면 해당 이미지 기반으로 서버를 실행해주는 형태
    • 요청 부하에 따라 자동으로 확정(Auto Scaling)

  • Object Storage

    • 다양한 Object를 저장할 수 있는 저장소
    • 다양한 형태의 데이터를 저장할 수 있으며, API를 사용해 데이터에 접근할 수 있음
    • 점점 데이터 저장 비용이 저렴해지고 있음
    • ML 모델의 Pkl 파일, CSV파일, Log등을 저장

  • Database(RDB)

    • DB가 필요한 경우, 클라우드에서 제공하는 DB 활용 가능
    • 웹, 앱 서비스와 DB가 연결된 경우가 많으며 대표적으로 My SQL, PosgreSQL등을 사용 가능

  • Data Warehouse

    • 서비스에서 활용할 DB
    • 일반 DB에 있는 데이터와, Object Storage에 있는 데이터 등을 모두 모아서 Data Warehouse 형태로 저장, Data 분석에 특화됨(쿼리가 더 빠름)

  • AI Platform

    • AI Research, AI Develop 과정을 더 편리하게 해주는 제품
    • ML Ops 관련 서비스 제공
    • Google clout platform TPU

    • 첫 가입시 300 크레딧 제공, 학습시 Cloud 비용의 부담이 적음

CI/CD

개발 환경

  • Local
    • 각자의 컴퓨터에서 개발, 환경 통일을 위해 Docker 등을 사용
  • Dev
    • Local에서 개발한 기능을 테스트할 수 있는 환경, Test 서버
  • Staging
    • Production 환경에 배포하기 전에 운영하거나 보안, 성능 등을 측정하는 환경
  • Production
    • 실제 서비스를 운영하는 환경 환경
  • 환경을 나누는 이유
    • 운영중인 서비스에 장애가 발생하면 안되기 때문

CI(Continuous Integration, 지속적 통합)

  • 빌드, 테스트 자동화

  • 새롭게 작성한 코드 변경 사항이 Build, Test 진행 후 Test Case에 통과했는지 확인

  • 지속적으로 코드 품질 관리

  • 10명의 개발자가 코드를 수정했다면, 모두 CI 프로세스 진행

CD(Continuous Deploy | Delivery, 지속적 배포)

  • 배포 자동화

  • 작성한 코드가 항상 신뢰 가능한 상태가 되면 자동으로 배포될 수 있도록 하는 과정

  • CI 이후 CD를 진행

  • dev, staging, main 브랜치에 Merge가 될 경우 코드가 자동으로 서버에 배포

→ 비용 문제로, 개인 프로젝트에서는 서버를 많이 두지 않는 경우도 존재

Backend Programing

Server 구성, Use Case

  • 실제 회사의 Use Case
    • App, Web Service Server가 존재
    • ML Service의 Server가 존재
    • Service Server에서 ML Server에 Inference를 요청 및 통신(or Service Server의 한 Process로 실행)

Server의 형태

  • Monolithic Architecture : 하나의 서버에서 모든 일을 처리
  • Microservice Architecture(=MSA) : 개별 서버를 구성하고 서로 통신

REST API

Representation State Transfer

  • Resource, Method, Representaton of resource로 구성
  • Create, Read, Update, Delete에 사용

URI, URL

  • URI(Uniform Resource Locator)
    • 인터넷 상 자원의 위치
  • URL(Uniform Resource Identifier)
    • 인터넷 상 자원을 식별하기 위한 문자열의 구성
    • URL을 포함함

HTTP Method

  • 종류

    • GET(read) : 정보를 조회하기 위해 사용
    • POST(create) : 정보를 입력하기 위해 사용
    • PUT(update) : 정보를 갱신하기 위해 사용
    • PATCH(update) : 정보를 갱신하기 위해 사용
    • DELETE(delete) : 정보를 삭제하기 위해 사용

  • GET vs POST

    처리방식GETPOST
    URL Data 노출 여부OX
    URL 예시localhost:8080/login?id=kylelocalhost:8080/login
    데이터 위치HeaderBody
    캐싱 가능 여부OX

Header와 Body

  • http 통신은 Request하고 Response를 받을 때 정보를 패킷에 저장
  • Packet 구조 : Header / Body
  • Header : 보내는 주소, 받는 주소, 시간
  • Body : 실제 전달하려는 내용

Status Code

  • 클라이언트 요청에 따라 서버가 어떻게 반응하는지를 알려주는 code
  • 1xx(정보) : 요청을 받았고, 프로세스를 계속 진행함
  • 2xx(성공) : 요청을 성공적으로 받았고, 실행함
  • 3xx(리다이렉션) : 요청 완료를 위한 추가 작업이 필요
  • 4xx(클라이언트 오류) : 요청 문법이 잘못되었거나 요청을 처리할 수 없음
  • 5xx(서버 오류) : 서버가 요청에 대해 실패함

동기와 비동기

  • 동기 : 서버에 요청을 보내고, 응답이 돌아와야 다음 동작을 수행
  • 비동기 : 서버에 요청을 보낼 때 응답 상태와 상관없이 다음 동작을 수행함

IP(Internet Protocol)

  • 네트워크에 연결된 특정 device의 주소를 나타내는 체계
  • 미리 약속된 IP
    • localhost or 127.0.0.1 : 현재 사용 중인 local pc
    • 0.0.0.0, 255.255.255.255 : broadcast address, 네트워크에 접속된 모든 장치와 소통하는 주소

Port

  • 해당 IP에 접속할 수 있는 통로
  • 0~65535
  • 0~1024는 표준 통신을 위해 미리 약속됨
    • ssh : 22
    • http : 80
    • https : 443

Fast API

FastAPI 소개 & 특징

  • 최근 떠오르는 python web framework
  • high performance : nodejs, go와 대등한 성능
  • easy : Flask와 비슷하며 micro service에 적합
  • productivity : swagger(문서) 자동 생성, pydantic(serialization) 사용 가능

FastAPI vs Flask

  • 장점
    • Flask보다 문법이 간결
    • 비동기 지원
    • Built-in API(Swagger) 지원
    • Pydantic을 이용한 serialization 및 validation
  • 단점
    • 아직 Flask의 유저가 많음
    • ORM 등 DB와 관련된 라이브러리가 적음

  • 구축된 프로젝트를 유지보수하는 경우

  • 기능

    • API 디자인

    • API 빌드

    • API 문서화

    • API 테스팅

FastAPI 기본지식

  • Path parameter
    • 서버에 path 형식으로 값 전달
      • /users/402
      • 데이터가 존재하지 않는 경우 : 404Error
  • Query parameter
    • 서버에 query string 형식으로 값 전달
    • api 뒤에 입력 데이터를 함께 제동하는 방식으로 사용
    • key-value 쌍으로 이루어지며 &로 연결해 여러 데이터를 전달할 수 있음
      • /users?id=402
      • 데이터가 존재하지 않는 경우 : empty list ⇒ 추가로 error handling 필요

Request Body & Response Body

  • Request body : send data client to server

    • 항상 데이터가 포함되는 것은 아님
      • GET의 경우 Request Header로 데이터 전달
    • 데이터를 보내고 싶다면 POST Method 사용

  • Response body : send data server to client

Pydantic

  • Data Validation / Settings Management 라이브러리

  • Type Hint를 런타임에서 강제해 안전하게 데이터 핸들링

  • 파이썬 기본 타입(str, int 등) + List, Dict, Typle에 대한 validation 지원

  • 기존 validation 라이브러리보다 빠름

  • config를 효과적으로 관리할 수 있게 도와줌

  • ML Feature Data Validation으로도 활용 가능

  • Validation Check Logic에 사용할 수 있는 방법

    • 일반 python class를 활용한 input definition and validation
      • 복잡한 검증로직이 필요한 경우 Class method가 복잡해지기 쉬움
      • Exception handling을 자유롭게 할 수 있지만 그만큼 손이 많이 감
    • Dataclass를(python ≥ 3.7) 활용한 input definition and validation
      • dataclass decorator 사용
        • init method가 필요하지 않음
      • __post_init__과 같은 편의 method 존재
        • validation 검증로직 작성용 method
        • 여전히 직접 작성해야 함
    • pydantic을 활용한 input definition and validation
      • 미리 정의된 validation class 존재
        • HttpUrl : 올바른 Url인지 검증
        • Field(ge=x, le=y) : 값이 x이상, y 이하인지 검증
        • DirectoryPath : 존재하는 디렉토리인지 검증
      • 어디서 에러가 발생했는지 location, type, message 출력

  • Pydantic Config

    • Config를 체계적으로 관리할 수 있는 방법을 제공

    • 설정 값을 상수로 코드에 저장 ⇒ Twelve-Factor 위반

      • Twelve-Factor : SaaS(Software as a service)를 만들기 위한 방법론을 정리한 규칙

    • validation처럼 BaseSetting을 상속한 클래스에서 Type hint로 주입된 설정 데이터를 검증할 수 있음

    • Field 클래스의 env 인자로 환경변수로 부터 해당 필드를 오버라이딩 할 수 있음

FastAPI 익숙해지기

  • Event Handler

    • 이벤트 발생 시 그 처리를 담당

    • 데코레이터로 손쉽게 subscribe 가능

  • Error Handling
    • FastAPI의 HTTP Exception : Error Response를 더 쉽게 보낼 수 있도록 하는 Class, client에게 더 자세한 메시지를 보내는 코드 작성 가능
  • Background Task

    • FastAPI는 Starlett라는 Async 프래임워크를 래핑해서 사용

    • 오래 걸리는 작업들을 background task에서 실행

    • online serving에서 cpu 사용이 많은 작업들을 background task로 사용하면 client는 작업 완료를 기다리지 않고 즉시 response를 받을 수 있음
